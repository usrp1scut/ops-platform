# Debugging Playbook

这份文档用于训练排错能力。目标不是更快猜答案，而是更稳定地把问题缩小到可以验证的范围。

## 基本流程

1. 复现问题：记录命令、请求、输入数据、预期结果和实际结果。
2. 明确错误类型：编译、启动、接口、权限、数据、并发、外部依赖或架构检查。
3. 缩小范围：从入口开始，沿调用链找到最后一个正常点和第一个异常点。
4. 建立假设：每次只验证一个假设。
5. 最小修复：只改导致问题的部分，避免顺手重构。
6. 验证结果：运行能覆盖问题的最小测试，再运行必要的回归检查。
7. 复盘记录：写下根因、误判、最终修复和下次如何更快定位。

## Go 后端问题

常用检查：

```bash
go test ./...
go build ./...
```

定位建议：

- 编译失败先看第一个错误，后续错误常常是连锁反应。
- handler 错误先确认 route 是否挂载，再确认 middleware、request decode、service 调用。
- repository 错误先确认 SQL、参数顺序、事务边界和 migration。
- 跨包调用异常时，对照 `docs/architecture/dependency-graph.md`。

## HTTP 和权限问题

定位顺序：

1. route 是否存在。
2. 请求方法是否正确。
3. token 是否有效。
4. middleware 是否要求了正确 permission。
5. 当前角色是否拥有 permission。
6. handler 是否把业务错误映射成了合适的 HTTP status。

常见现象：

- `401` 通常先查认证。
- `403` 通常先查 RBAC permission。
- `404` 先查 route 和 path 参数。
- `500` 先查日志、数据库错误和未处理分支。

## 数据库问题

定位顺序：

1. migration 是否包含目标表、字段、索引和约束。
2. SQL 是否和 schema 对齐。
3. 空值、唯一约束、外键约束是否被正确处理。
4. 事务是否覆盖了必须原子完成的操作。
5. 测试数据是否污染了后续测试。

## Worker 和外部依赖问题

先区分问题来自内部逻辑还是外部依赖：

- AWS sync 先确认账号模式、凭证来源、region 和 SDK 返回错误。
- bastion probe 先确认 target、proxy、超时、协议连接和 cleanup。
- OIDC 问题先确认 issuer、client id、redirect url 和用户同步逻辑。
